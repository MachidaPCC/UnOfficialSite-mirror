おうちです。ホントすみません >> 何か悪いことをしたかと気がかりです

悪口を言われたとかではなく、指摘されていた「日記を書いていない」という怠惰を七大罪に誇張拡張して（大馬鹿）書きました。人を悪く見せるようなことを書いてしまって、本当に申し訳もない。


忘れる前に日記検索ページについて書いておきます。

作るにあたって、先駆者の HRtP さんが試された 2 つの方法を改めて検討しました。ウェブサイトで標準的に使える検索要素と google 提供の検索機能です。前者はサーバーにリクエストを送って検索するものなのですが、このサイトは忍者ホームページというサービスを使って運営されているので、無料なのもあり、サーバーサイドのプログラムを動かすことができません。これは、クライアントから指令が送られサーバー側でプログラムを実行し結果を返すというサーバー側の負担が大きい代物なので、このようなレンタルサービスでは使えないのも当然です。後者は原因がいまいちわかりませんでしたが、おそらく google の検索エンジンにこのサイトの各ページがあまり認知されていないのかな、という感じです。文字コードが sjis なのももしかすると良くないのかも知れません。

なので、クライアントサイドプログラミングで実装することにしました。プログラム本体をクライアント側に送り、そちらで実行させる仕組みなので、ブラウザーが javascript に対応してさえいれば問題なく動くはずです。過去にはガラケーやら Opera やら IE やら色々なブラウザーがありそれぞれに対応しなければならなかったようですが、chromium にほとんど統一されてしまった現在では特段気にする必要もないでしょう。

javascript では、同一サイト内ならサーバーにリクエストを送ってファイルを貰うことができます。はい。日記の html 文書を丸ごと持ってきて検索するという力技です。3 つほど問題があったのでそれぞれ話します。

まず、先に言ったとおり文字コードが shift-jis であること。javascript は標準で utf-16 を使うので、ファイルをそのまま読み込むと文字化けします。さらに、javascript で用意した検索結果の文字コードと検索ページの文字コードも一致させなければなりません。この辺りがややこしくて大分時間を使いました。幸いにも javascript に sjis のデコード機能が標準搭載されていたので、日記（sjis） -> js (utf-8) -> 検索ページ (utf-8) の流れで、ファイルを直接読み込まず、バイナリーを受け取って一回だけ手動でデコードすれば良くなりました。将来、日記の文字コードが utf-8 になるようなことがあれば、ファイル先頭だけ読んで文字コードを特定し読み直すという周りくどい方法が必要になると思います。いやだな。

2 つ目、検索結果の表示方法に迷いました。google 検索などで、検索先のページで単語がハイライトされていることがありますが、始めはあれ（テキストフラグメントと言います）を利用していました。しかし、ブラウザーの仕様によって結果がまちまちだったり、文脈の指定が面倒だったり、色々と問題があるので結局不採用。文脈は検索ページで表示し、リンク先は各日記の先頭という仕組みに作り変えました。

最後は検索速度と並べ替えの両立。というか、javascript をあまり触ってこなかったので、そもそも同期・非同期の処理の扱い方が分からず苦労しました。というのも、サーバーにリクエストを送る関数は、ファイルが返ってくるまでの間に他の処理ができるように非同期となっているのです。ちょっと説明が難しいですが、実行すると空の値が返ってきて、ファイルが帰ってくると返値が更新されるというイメージです。非同期処理のまま一斉にリクエストを送ると、順番に関係なく処理が終わったものから順次帰ってくるので、後で並べ替える必要が出てきます。しかし、非同期処理をいちいち同期的に実行すると、絶望的な速度になってしまいます。どうしたものか。結局前者を採ったのですが、そこで javascript の仕様をうまく使えたので我ながら自慢に思っています。javascript のオブジェクトは、キーとバリューがセットになった辞書式の情報なのですが、このキーが数字のみのオブジェクトは、中身の順番を自動で並べ替えてくれるようになっています。リクエストを送り、帰ってきた結果を日付をキーにしてオブジェクトに追加すれば、並べ替えとかいう非効率な処理をすっ飛ばすことができるわけです。いや、内部的には実際並べ替えられているはずですが、低レベルな場所で行われる処理の方が効率的に決まってます。これは賢い（やかましい）。

というわけで、他にも色々面倒はあったのですが、実は似たようなこと（ファイルサイズの取得）を試している最中だったのもあり、比較的すんなり作れて、あーよかった。テスト前だったこと、勉強時間を著しく削ったことを除けば完璧です。←おい。

というわけでみなさん、検索しましょう。色々試せて楽しいですよ！　検索対象を本文だけでなく記者・曜日にも設定できるので、正曜日とか翠曜日とか言ってる人を絞り込むこともできるわずです。


では、私は今日の昼飯担当に任ぜられてしまったのでこれから買い出しです。さいなら。

いつか話した「本格ミステリ01」を紹介します。本格ミステリ作家クラブが年に 1 度本格ミステリ大賞を出しているのですが、その選考に入らない短編作品や評論を集めたのがこのアンソロジーです。単行本未収録という条件と評論を含めるという点が他に類を見ず面白いところ。残念ながら、この企画は後に「ベスト本格ミステリ」「本格王」と名前を変えて縮小していくのですが。

とまれ、2000年発表作品の傑作選である本作は量も質も申し分ありません。2 段組でたっぷり 654 ページ、仰向けで読むのが辛くなってくる重さです。では、ここからは各収録作品の紹介に移ります。短編 13 編、評論 3 本です。

紅雨荘殺人事件／有栖川有栖
有栖川はこのクラブの会長です。私は何故か有栖川作品はあまり読んでいないのですが、安定して面白い作家。これは火村の方、作家アリスのシリーズ。どこかの解説で証拠の見せ方は 2 種類あると読んだのが思い出されました。1 つは、証拠を明示してそれが何を意味するかを問うもの。もう 1 つは、さりげない描写が実は証拠になっているというもの。本作で言えば指紋が前者にあたりますが、そこから犯人へと推理を繋げるには後者の隠された証拠に気付く必要があります。トリックから話を考えたというだけあって秀逸です。その他、犯人の不可思議な態度という一捻りが加えられているのが見どころ。
鳥居の赤兵衛／泡坂妻夫
泡坂妻夫大好き。「亜愛一郎」シリーズや本人が奇術師であったことがよく知られています。本作の謎自体は途中で想像つくようなものですが、ペンネーム自体がアナグラムな作家らしい名前遊び、捕者・帳という作品性が楽しめれば面白い一品。
四角い悪夢／太田忠司
新しく読んだ作家の短編では一番面白かった。妹が見た悪夢の内容を兄が推理するという内容ですが、問題を「何を夢に見たのか」から「何故そんな夢を見たのか」にシフトさせる手法が実に鮮やかです。ライトな人物造形に対して全体に漂う哀しい雰囲気も巧みで私好み。是非他の著作を読みたいです。
子供部屋のアリス／加納朋子
早期退職の後に探偵を始めた男と若い押しかけ助手のコンビを主人公に据えたシリーズの一作。以前触れた人物の呼び方が親切な方の作品です。所謂「日常の謎」系で、細かな手掛かりの散りばめ方が良くできています。ただ、報酬を受け取って探偵をやっている癖に契約を破って行動しているのは職業倫理に反していてノイズでした。別に登場人物が皆潔白である必要はないのですが、視点人物の価値観はある程度現実に合わせて貰わないと読みづらくなってしlまうと思うのです。
邪宗仏／北森鴻
両腕がない仏像の出所と殺人犯を並行して推理する作品。景教関連だというので、ちょうど前回読んだ作品と繋がる内容です。江戸時代におけるキリスト教は鎖国と禁教の影響で仏教と混じったとかそんな話なのですが、「沈黙」はあくまで当事者の目線で語っていたのに対し、こちらは民族学研究者の考察という体なので幾分かわかりやすい。死体の腕が仏像と同じく落とされているという状況で何故かエラリー・クイーンを思い出し（エジプト十字架？）、実は死体の腕が切り取られた後に仏像が加工されたのだという転回を思いついたものの、数ページで否定されていて自分の推理能力の NASA に失望。あとは、探偵役のキャラクターがいかにも・・・・な女性像なのがむかついたくらいで、作品自体は良くまとまっていました。
人を知らざることを憂う／鯨統一郎
解説で言及されているとおり「バカミス」に分類されるような作品ですが、倉阪鬼一郎や鳥飼否宇に比べるとバカさ加減が足りていない。そこで張り合う必要は微塵もないので良いんですけどね。密室とダイイングメッセージが主眼。長すぎるダイイングメッセージと乗犬というバカバカしい要素や本格もののパロディで重要な証拠を隠蔽する「隠す証拠」タイプです。私もまんまと騙されてしまいました。
正太郎と井戸端会議の冒険／柴田よしき

黒の貴婦人／西澤保彦

つまらない話は適当に切り上げて、別のつまらない話をしましょう。第 5 回音声入門です。

皆さんはいわゆるハイレゾ、ハイレゾリューション音源をご存知でしょうか。粗大ゴミです。私はこのハイレゾをチヤホヤする風潮が大嫌いなので、ハイレゾには意味がないことを数回かけて説明していきます。といって専門知識があるわけでもないので、Xiph.org に大変お世話になる……というか、ほとんど某記事の要約になるかもしれません。英語の記事なのでここに広く紹介する意味はある、はず。今回は容量と周波数の話です。

そもそもハイレゾリューションは直訳すれば「高解像度」、要するに従来の CD 規格を超える解像度の音源のことを言います。音声データにおける解像度は量子化ビット数と標本化周波数の積であるということは、第 2 回で話したので参照のこと。CD は 16 bit / 44.1 kHz、多くのハイレゾは 24 bit / 192 kHz です。確かに解像度はとても高くなっています。しかし、非圧縮における 1 秒あたりのビット数を比べると、

16 bit * 44.1 kHz * 2 = 1411.2 kBit / s
24 bit * 192  kHz * 2 = 9216  kBit / s

ハイレゾは CD の 6.5 倍にも上ります。私がｱｲﾌｫｰﾝに入れている音楽 10 GB が 6 倍になったらストレージが 4 GB しか残りません。

さらに、ハイレゾは数字の上では高解像度ですが、実質的には CD 音質を越えられません。人間が聞き取れる周波数は 20 Hz ~ 20 kHz と言われていますが、これは周波数ごとの「聞き取れる最小の音量」「痛みを感じる始める音量」が試験された結果です。20 kHz の音はどの年齢でも両者が 140 dB 程度になり、この大きさの音には聴覚機能に異常をきたす危険性があります。20 kHz というのは個人差や年齢差があるなかで最も寛大なラインを取った値なのです。標本化定理によれば、標本化周波数が 44.1 kHz なら理論上は半分の 22.05 kHz の音までは完全に復元できます。192 kHz は鑑賞目的には過剰です。

可視光と可聴音を比べてみましょう。TV に使う IR リモコンを思い浮かべてください。近赤外線で通信していますが、まさかこれが見えるという人は居ませんね。可視光線の周波数は 400 THz ~ 800 THz 程度、近赤外線の周波数は 300 THz 程度です。両者がどのくらい離れているのかを計算すると、

|400 - 300| / |400 - 800| = 0.25

近赤外線は可視光線の周波数を 25 % 越えています。いっぽう、可聴音の周波数を 20 Hz ~ 20 kHz、ハイレゾ音源のナイキスト周波数を 96 kHz として計算すると、

|20000 - 96000| / |20000 - 20| = 3.803……

ハイレゾで再現される音は可聴音を 380 % 越えています。これが無駄でなくて何だと言うのでしょう。無駄なんだ……無駄だから嫌いなんだ無駄無駄……

  • 可聴音は 20 kHz、CD は 22 kH まで。ハイレゾの 96 kHz は過剰

でした。今回はハイレゾは無意味どころか有害ですらあることを説明します。

人間の耳に聞こえないような高周波数の音は、スピーカーで再生する際にノイズの原因になります。アナログ機器の非線形性のせいで生じるこの歪みを、相互変調歪み intermodulation といいます。素人なりに色々調べてみているのですが、難しい数式がジャラジャラ出てきて理解不能なので、理解した（と思しい）基本的な原理だけここに書きます。

相互変調歪みの前に単一の周波数の音に発生する歪みを考えましょう。非線形性は、入力に対して出力が線形でないという特性です。スピーカーの場合、入力電圧に対して出力電圧がだんだん小さくなります。つまり、入力電圧が大きくなり続けるとやがて出力電圧が追いつかなくなってしまいます。この非線形性は全高調波歪み total harmonic distortion を生みます。50 kHz の波なら 150 kHz、250 kHz、350 kHz、450 kHz……の高調波が発生します。これは元の倍数の周波数で、harmonic というとおり必ずしも不快な音ではありません。倍音歪みというやつで、人の耳はこれを元の音 1 つとしてまとめて認識しやすい仕組みになっているらしいです。

しかし、入力の音が 2 つの周波数を持っていたらどうなるでしょう。まず、それぞれの周波数に対し調和な歪みが発生します。そして、2 つの元の音や全高調波歪み同士が互いに影響し合って非調和な歪みが発生します。問題は、この相互変調歪みが不協和音であるということです。単一の周波数ならば協和音的な歪みしか発生しませんが、複数の周波数となると混ざり合って無関係な音が生じてしまいます。1000 Hz と 1100 Hz の音があったとすると、まず 2000 Hz、2200 Hz といった元の音と調和な歪みが生じます。次は、100 Hz と 2100 Hz という不調和な歪みが生じます。周波数において乗算は調和的ですが、和算と減算は非調和的です。さらに悪いことに、第 2 第 3 の全高調波歪みは第 2 第 3 の相互変調歪みを生みます。200 Hz と 4200 Hz、300 Hz と 6300 Hzといった具合に。

では、歪みについて理解したところでハイレゾに戻りましょう。この手の不快な歪みは、低／高周波数で特に大きくなります。つまり、超高音域を収録したハイレゾは歪みも大きいということです。30 kHz と 33 kHz の音を再生すると、可聴域を含む広域にわたって耳に聞こえるレベルの歪みが発生しました。（いちおうリスニングテストで聞き分けられるとの結果も出ていますが、これはちょっとアヤシイです。）もちろん、通常のスピーカーはこのような歪みを考慮した上で設計されていますが、超高音域の音を再生されれば話は別です。あくまで聞こえる音を快適に聞かせる機械ですから。

スピーカーを超高性能にするとか、頑張ってノイズを低減するとか、そういった解決策も考えられます。しかし、端から聞こえない音に対してそこまでのコストをかける必要性がありますか？　このような対策を行なっても可聴音への影響は避けられません。すると、最も現実的な対応は「そもそも高音域をエンコードしない」つまりハイレゾを捨て、高周波成分を切って、CD レベルで記録するという方法です。

ハイレゾは嵩張るだけでなく再生にネガティブな影響を及ぼします。CD と変わらないというだけでなく、CD の方が優れているのです。標本化周波数は大きくとも 48 kHz、特に拘らなければ 44.1 kHz で十分だとお分かりただけたことでしょう。次回はビット深度に話を進めます。

written by おうち

おうちです。先日の発言に思うところがあったので工作をしています。何をしているかはまだ秘密ですが、このサイト上にあるので暇人各位は探してみてください。いや、見つかると困るのでやっぱり探さないで……。やる気が溢れて水木金と 1 日 10 時間くらい費やしたんですが、土日は作業できず手持ち無沙汰なので日記を書こうと思った次第。宿題にもそろそろ手をつけないといかんな。

で、今日はまたハイレゾの話です。予告通りビット深度の観点から述べます。

復習。ビット深度は振幅の量子化の細かさを表し、量子化ビット数とも言います。値が大きいほど詳細な信号が得られ、小さいと表せる音量の幅（ダイナミックレンジ）が狭まったりノイズが現れたりします。

誤解されやすいため音量の単位についても確認しましょう。よく「爆発音は 120 dB」などと言います。しかし、dB そのものは比を表す相対値であり、絶対的な物理量を表す値ではありません。音圧や電力においては、この dB に基準となる値を与えることで絶対値として使っているのです。（素人の私などは、単位をきちんと付けてくれた方が明確で良いのにと思います。）

少々込み入った話なのでベル B から順番に説明します。基準値と比較対象の比の常用対数がベル B です。0 B を基準として、1 B は 10 倍、2 B は 100 倍、-1 B は 1 / 10 倍、 -2 B は 1 / 100 倍です。このままではあくまで相対値ですが、例えば 1 秒を 0 B s と決めると値が意味を持ちます。このとき、1 B s は 10 秒、1 分は約 1.78 B s です。

で、デシベル dB はベルの単位を 1 / 10 にしたものです。10 dB が 0 dB の 10 倍、20 dB が 0 dB の 100 倍。逆では無いので、注意。1 秒を 0 dB s としたとき、-1 dB s が約 0.79 秒、-10 dB s が 0.1 秒です。

これでようやく音圧の dB の話ができます。音圧の dB の基準は Sound Pressure Level、SPL です。というか、先ほどから使っている「音圧」が SPL の訳語です。SPL の単位は 気圧 Pa で、0 dB SPL は 20 μPa と定められており、これは人間が聞きとれる最も小さな 1 kHz の音の音量です。ただし、音の強さは音圧の 2 乗に比例するので、

音圧 Lp (dB) = 10 log10 p2 = 20 log10 p

です。これでようやく実態がわかりましたね。「爆発音は 120 dB」だと言われたとき、この 120 dB は 

20 μPa * 20 log10 120 = 20000000 μPa = 20 Pa

です。

では、人間の耳に聞こえる音について調べてみましょう。最も小さい音は定義からして 0 dB……かと思いきや、周波数によって聞こえ方が異なるため、2 kHz～4kHz では -8 dB まで聞こえるそうです。大きい方はせいぜい 110 dB。十分に若ければ 140 dB という話もありますが、そんな音を聞き続けていれば耳がいかれてしまいますし、音楽鑑賞と言う点では問題外です。聞こえる範囲は大きく見積もって 120 dB というところでしょう。

ここで、CD のビット深度は 16 bit、ダイナミックレンジ（信号の最大と最小の差）は 6 倍して 96 dB です。おや、100 dB もありませんね。ハイレゾは 24 bit で 144 dB です。では、CD だとダイナミックレンジが足りず、ハイレゾなら十分なのでしょうか。いや、そんなことはありません。

まず、聞こえる音の最低値を見直します。先ほどは -8 dB と言いましたが、これは全く無音の環境を想定した値です。実際には、プロの収録スタジオにすら 20 dB の環境雑音があります。ちなみに 100 W 白熱電球のハムノイズが 10 dB です。無響室のように全く音の無い部屋に入れば自分の鼓動や血潮の音が聞こえてしまうのだから、雑音を排するのは不可能です。

さらに、16 bit の理論上のダイナミックレンジは 96 dB ですが、実際にはもっと大きいはずです。量子化の誤差によって生じる歪みをノイズフロアといい、ダイナミックレンジの信号の最小の値はこのノイズフロアで決まります。量子化によって生じる一定の誤差がノイズとなり、ノイズフロア以下の小ささの信号は埋もれてしまって聞こえないという寸法です。しかし、先も人間の耳は 1 kHz で 0dB、2～4 kHz だと -8 dB まで聞こえると言いましたが、人間の耳は全ての周波数の音を均一に聞く訳ではありません。ですから、ノイズフロアの実効値は実際よりも小さくなるはずです。さらにディザリングなどノイズを減らす工夫をすれば 96 dB から 120 dB までダイナミックレンジを広げられるといいます。分かりづらいですがこれは実に 16 倍以上に相当します。

長くなりましたが（そして、後半は駆け足になってしまいましたが）ビット深度の話はこれでおしまいです。書いた文章が途中で半分くらい消し飛んだのもあって疲れました。しかし、標本化も量子化も CD レベルで十分だと言うことがよく理解していただけたことと思います。次回は収録の話、次次回は良い音を聞くための話、それくらいでこのコーナーも終わりにしようかな。ではではﾉｼ

/(?=.*(<dl>.*?<\dl>))\1/gs

S-JIS だとバックスラッシュが文字化けるので円記号で勘弁。高さを稼ぐために 1 文字ずつ解説！

/ ... /gs
スラッシュで囲むのは正規表現であることを示すため。後ろはフラッグで、g は global、s は single line。後述。
(?= ... )
先読み。後ろに続く文字を指定する。
.*
. は任意の 1 文字にマッチ。single line がついているので改行にもマッチする。* は直前の文字を 0 回以上繰り返し、最長でマッチする。2 つを組み合わせると、なるたけ沢山の任意の文字になる。
( ... )
キャプチャグループ。今回はマッチを再利用するために使った。
<dl>, <\dl>
文字列そのまま。
.*?
? をつけると最短一致に変わる。.* は最右最長、こちらは最左最短。
\1
1 番目のキャプチャグループを再利用。今回は <dl>.*?<\dl> に相当する。

これで何が起こるかというと……

まず、先頭に先読みが来ているので empty string がマッチします。次に、先読みの中身が文書の先頭から最後の <\dl> までにマッチします。その中で 1 番目のキャプチャグループが最も後ろの dl 要素にマッチします。これを先読みの後ろで再利用。

結果として、empty string + 最後の dl 要素、つまり最後の dl 要素がマッチします。終わり。わけわかめって人は regex 101 で遊んでみてください。慣れると面白いですよ。
